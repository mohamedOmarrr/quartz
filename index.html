What is a
Background Service?

Long-running processes that keep your system running — even when no one is watching.

A Background Service is a long-running process designed to execute tasks independently of user interaction. It is request-agnostic — meaning it does not rely on incoming HTTP requests or UI triggers to function. Instead, it operates continuously in the server's background, maintaining its own lifecycle to perform recurring or deferred operations.
Use Cases

Practical Examples

EXAMPLE 01

⏰ Automated Subscription Expiration Notification

The service periodically checks the database for users whose subscription has expired or is about to expire, comparing the EndDate with the Current Date — then fires off reminder emails automatically.

EXAMPLE 02

🔄 Periodic Cache Refresh

A background task clears or refreshes the system's memory cache at specific intervals to ensure data consistency across the application without manual intervention.

EXAMPLE 03

🗃️ Database Log Archiving

To maintain high performance, the service automatically moves old logs (older than 30 days) from the active table to an archive table — keeping queries fast and storage clean.

💡 Key Takeaway
My project needs a Persistent Server.
To guarantee that the Background Service process is always running, we must deploy on a server that maintains a continuous, live process — not a platform that kills the process after each request. A serverless platform cannot keep a Quartz.NET scheduler alive between triggers.
📝 Note — Alternative Approach
It is still possible to use Serverless Task Scheduling as an alternative. For example: keep Quartz.NET's schedule definitions, but host the actual job logic inside an Azure Function (or AWS Lambda). The function gets triggered on a timer, executes the logic, and exits. This works for simple, short-lived tasks — but for continuous observers or stateful jobs, a persistent server remains the right choice.
Architecture

Why Use a Dedicated Worker Service?

While you can run background tasks inside your main web application, creating a separate Worker Service project (within the same Solution) becomes important as your system grows.

🛡️

Fault Isolation — Crash Resilience

Running background tasks in a separate process ensures that any errors or crashes within your background jobs do not bring down your main web application. Your users experience uninterrupted service even if a background job fails hard.

📈

Resource Management & Independent Scalability

Heavy background tasks (report generation, image processing) can be CPU/RAM intensive. Isolating them prevents resource competition with the web app, and allows you to scale the worker independently based on task load.

⚠️ Important Note for Smaller Projects
If your application is small or has a limited number of background tasks, don't over-engineer it. You can safely host Quartz.NET jobs directly inside your main web application process — this reduces infrastructure complexity and deployment cost. Only move to a dedicated Worker Service once your system grows in complexity, load, or requires higher reliability.
Library

What is Quartz.NET?

Quartz.NET is a Job Scheduling Library that links Jobs (your business logic) to Triggers (your schedules), ensuring your code executes automatically at precise times — without any manual intervention.

Register all your jobs inside Program.cs:

Program.cs — Quartz Registration 
builder.Services.AddQuartz(q =>
{
    // ── Job 1: Cleanup Job ──────────────────────────────
    var cleanupJobKey = new JobKey("CleanupJob");
    q.AddJob(opts => opts.WithIdentity(cleanupJobKey));
    q.AddTrigger(opts => opts
        .ForJob(cleanupJobKey)
        .WithIdentity("Cleanup-Trigger")
        .WithCronSchedule("0 0 1 * * ?"));   // runs at 1:00 AM daily

    // ── Job 2: Email Reminder Job ───────────────────────
    var emailJobKey = new JobKey("EmailReminderJob");
    q.AddJob<EmailReminderJob>(opts => opts.WithIdentity(emailJobKey));
    q.AddTrigger(opts => opts
        .ForJob(emailJobKey)
        .WithIdentity("Email-Trigger")
        .WithCronSchedule("0 0 9 * * ?"));   // runs at 9:00 AM daily
});

builder.Services.AddQuartzHostedService(q => q.WaitForJobsToComplete = true);

Place the Job class inside your BLL Layer with its own logic:

BLL / CleanupJob.cs — Job Class 
public class CleanupJob : IJob
{
    private readonly GymDbContext   _db;
    private readonly ILogger<CleanupJob> _logger;

    // Inject DbContext and Logger via constructor
    public CleanupJob(GymDbContext db, ILogger<CleanupJob> logger)
    {
        _db     = db;
        _logger = logger;
    }

    public async Task Execute(IJobExecutionContext context)
    {
        _logger.LogInformation("CleanupJob started at: {time}", DateTime.Now);

        // Delete subscriptions that expired more than 1 year ago
        var oldDate          = DateTime.Now.AddYears(-1);
        var oldSubscriptions = _db.Subscriptions
                                   .Where(s => s.EndDate < oldDate);

        _db.Subscriptions.RemoveRange(oldSubscriptions);
        await _db.SaveChangesAsync();

        _logger.LogInformation(
            "CleanupJob finished. {count} records removed.",
            oldSubscriptions.Count());
    }
}
